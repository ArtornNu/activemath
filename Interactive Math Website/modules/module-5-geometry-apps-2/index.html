The big idea, before any activity

Most students meet a matrix as a rectangle of numbers with a multiplication rule that feels arbitrary. This session flips the order: students meet motion first, notice a pattern, and only afterward see that the pattern has a name and a notation.

👩‍🏫 Facilitator note

🤔 Quick check-in

In your own words: what's the difference between meeting a pattern first vs. meeting a rule first?

A pattern met first is something you own — you can re-derive it if you forget the name. A rule met first is something you either remember or don't. This module is a bet that the first kind sticks longer.

Activity 1 — Familiar Moves, New Notation

Before anything new: three transformations you already know — translation, rotation, reflection — and a new way to write them down.

Writing points as a matrix

A point (x, y) is one column. Three points side by side become a matrix with three columns — one column per point. For P₁(1, 1), P₂(3, 1), P₃(1, 3):

How to read it: the top row is every point's x-coordinate, in order (1, 3, 1). The bottom row is every point's y-coordinate, in order (1, 1, 3). Each column, read top-to-bottom, is one point — column 2 reads (3, 1), which is P₂. Try it yourself: what point does column 3 give you?

This is a points list — a tidy way to list points, nothing is being "done" to them yet. Later activities use a different kind of matrix, called the machine: it doesn't just sit there like this one — it moves every point somewhere new. Same bracket notation, two different jobs — worth keeping straight.

👩‍🏫 Notice

Rotation and reflection here pivot around the origin (0, 0) — that's why the whole triangle can jump to a totally new spot, not just spin or flip in place. Translation is the only one of the three that does not keep the origin fixed.

👩‍🏫 Advanced insight — for teachers

Translation uses addition because it isn't a linear transformation — the origin itself moves. Rotation and reflection keep the origin fixed, so each can be written as multiplying every point by a fixed 2×2 matrix. You'll build exactly how that multiplication arithmetic works in Activity 4 — for now, just notice that a 2×2 bracket of numbers shows up either way.

🎯 Target matrix

The target was built from [[1, −1], [1, 1]] — notice the roof peak and both base corners all sit exactly on grid intersections. That's on purpose: use those crossing points as your alignment marks next time.

📛 New words, now that you've played

The two arrows you were dragging are usually written î and ĵ. The grid they create — and everything sitting on it — is called the effect of a matrix. You just built one by hand.

Careful — this is a different job for the word "matrix" than Activity 1's points list. There, matrix meant a list of points, sitting still. Here it's the machine — it moves every point somewhere new, by setting up new reference lines (î, ĵ) instead of nudging each point one at a time. Same brackets, two different jobs.

🤔 Notice something?

You only ever dragged two arrows — yet all ten points of the house (roof, walls, even the little door notch) landed in exactly the right new spot, automatically. You never touched most of those points directly. How is that possible?

Here's a clue: a point on the house was never really stored as "so many steps right, so many steps up" on a fixed standard grid. It was stored as "so many steps along î, so many steps along ĵ" — and that recipe never changes, no matter where you drag î and ĵ to. What changes is only where the recipe points to, because î and ĵ themselves moved.

So every point actually has two addresses at once: one counted in î/ĵ-steps (which never changes here), and one on the standard grid (which changes whenever î/ĵ move). Activity 3 asks that question directly, for a single point: read the very same point's address off two different rulers — do you get the same numbers?

You only ever dragged two arrows (î, ĵ) — yet all ten points of the house moved correctly. Why?

👩‍🏫 Advanced insight — for teachers, not necessarily for your students at this depth

What you've been calling "the frame built from the red and purple lines" has a technical name: a basis. There is one point D in the plane; (1, 3) is its coordinates relative to the new basis; the address you just logged on the standard grid is its coordinates relative to the standard basis. Moving from one to the other is called a change of basis.

We've deliberately kept the bracket-style matrix notation off this activity — it hasn't been read or defined yet. The red-line/purple-line positions you've been logging are exactly the two columns of that notation; Activity 4 is where the compact written form actually gets introduced.

D's address on “my frame” never changed while you dragged the red/purple lines. Why not?

Activity 4 — Where the Formula Comes From

This machine's arrows are fixed — you can't drag them. Instead, drag the free point anywhere and watch its image. Before checking, try predicting.

🔮 Drag the free point — predict before you look
point = (1, 1) → image = (2, 1)
Step 1 — gather several casesClick each button below (or drag the point yourself) and watch each case log underneath. The more cases you collect, the clearer the pattern gets.
Step 2 — predictBefore clicking (3, −1) above, calculate where you think it will land. Then check against the log.

💡 The rule

This machine is [[1, 1], [0, 1]]. Every point becomes (x + y, y) — that's new_x = 1·x + 1·y and new_y = 0·x + 1·y. Read each as "row of the matrix, dotted with the point" — that's the origin of row-times-column multiplication.

🌍 Row-times-column isn't just a classroom rule — it's the exact operation behind rotating a photo, resizing an image, and warping a character on screen, done billions of times a second.

For the machine [[1, 1], [0, 1]], where does (2, 3) land?

💡 The formal name

This is the determinant. Its sign tracks orientation; its size tracks the area scale factor.

🌍 Checking det(M) before solving anything is like checking there's fuel before a trip. Activity 8 uses exactly this — one number says instantly whether a system even has one clean answer, before you solve it.

↩️ Drag until green matches the plain square

Before you startWhy does the green shape start off-square at all? (It's the unit square, already transformed by the fixed M.)
While draggingOnce green sits exactly on the orange target, your current arrows describe a matrix. What relationship do you expect it to have with M? Watch the det readout above as you go.
Before you revealWhat do you think would happen if M's det were 0 — could you ever drag your way back to a plain square? Try to answer before reading on.

💡 The inverse

M⁻¹ = (1/det(M)) · d−b−ca = (1/1) · 1−1−12 = 1−1−12. Applying M and then M⁻¹ (exactly what you just did by hand) brings every point back to where it started — that's what "inverse" means.

🫓 A matrix with no inverse

Try [[2, 4], [1, 2]] — det = 2·2 − 4·1 = 0. The unit square flattens to a line, so many different points get squashed onto the very same output. There's no way to "undo" that and recover which input you started from — so no inverse can exist. That's why det = 0 always means "not invertible."

🌍 This “undo” move is exactly how GPS correction and decompressing an image work: solve the inverse once, then instantly reverse-engineer any output back to its input.

If det(M) = 0, can M ever have an inverse?

No word problem yet — just the bare numbers. We'll build up from the smallest possible case to the exact system Activity 9 asks you to solve, one extra piece of generality at a time.

👩‍🏫 Facilitator note

This is the simplest version of everything in this whole module: an unknown scale, applied to something fixed, aiming at a target. Every later idea — basis vectors, matrix multiplication, systems of equations — is a generalization of exactly this one question.

Step 2 — one equation, TWO unknowns
(x, y) = (1, 1)  →  2x + y = 3 (need 8)
Now add a second unknown2x + y = 8. Drag the point until 2x + y really does equal 8. Keep going — notice there's more than just one place that works.

👩‍🏫 Facilitator note

Your students already solved 2×x=8 — one unknown, one answer. Add a second unknown to the same kind of equation, and the answer set stretches from a single point into a whole line. Same idea, one more dimension — that's what generalizing looks like. (4, 0) from Step 1 still works here too; it's just one point on this line, no longer the only one.

Step 3 — a SYSTEM of two equations
(x, y) = (1, 1)
Add a second equationNow also require x + y = 5. Drag the point until BOTH lines pass through it at once — where the two lines cross is the only (x, y) that satisfies both equations.

👩‍🏫 Facilitator note

Nothing new here mathematically — this is the method already in every textbook: graph both lines, read off where they cross. But notice: the second equation is what shrank "a whole line of answers" down to "exactly one answer." One more constraint, one fewer degree of freedom — that pattern is exactly why matrices can scale to any number of unknowns later, even past what a picture can show.

current guess (x, y) = (1, 1)

👩‍🏫 Facilitator note

You just solved the exact same system two completely different-looking ways — intersecting lines, and combining vectors — and landed on the identical answer both times. That's not a coincidence, and it's not two separate topics to memorize. It's one structure, seen from two angles. Noticing that is what doing mathematics actually feels like.

💡 Why bother with the matrix at all?

Graphing two lines works fine for 2 unknowns — so what does the matrix actually add? Two things a picture alone can't give you: det(M) — one number that answers "exactly one solution?" with no drawing at all, and works the same whether there are 2 unknowns or 200. M⁻¹ — solved once, it re-solves the very same system instantly for any new target, no re-graphing needed. Both are the same move as everything above: take something you can already check by eye, and turn it into a number or formula that keeps working long after the picture runs out of dimensions to draw in.

🔀 Two ways to read the same multiplication

Activity 4 read each new coordinate row by row — "a row of M, dotted with the point." Here we'll read the very same multiplication a different way: column by column, as x copies of column 1, plus y copies of column 2. Both readings always give exactly the same numbers; row-reading is quick for computing one coordinate at a time, column-reading is what makes "how much of each ingredient reaches this total" problems visible as a picture — like the dashed path you just dragged toward the target above.

By pictureDrag the input point until the output lands exactly on the target crosshair. Read off (x, y) — that's your answer.

💡 By formula

M⁻¹ = (1/det(M)) · 1−1−12 (same as Activity 6, det(M) = 1). So (x, y) = M⁻¹·(8, 5) = (1·8 + (−1)·5, (−1)·8 + 2·5) = (3, 2). One pencil = Rp 3,000, one eraser = Rp 2,000 — matching wherever you landed by dragging.

🌉 The visual ↔ algebra bridge

Dragging the input point until it lands on the target is the same act as multiplying both sides of M·(x,y) = (8,5) by M⁻¹ — both are ways of "undoing" M. The picture doesn't approximate the algebra; it is the algebra, just performed by hand instead of by formula.

📐 Why matrices — and why this scales

The same idea works for 3, 4, or 50 unknowns: name them, translate each sentence into one row of coefficients, stack the rows into a bigger matrix, and multiply by its inverse. The only thing that stops working past 2–3 variables is the picture — we can't easily draw a 4-dimensional arrow. But the algebra keeps working exactly the same way, which is exactly why it's worth learning the algebra and not just the picture: the picture builds the intuition, the algebra carries it further than a picture ever could.

Reconnecting to the textbook

👩‍🏫 Unlocked

You've now opened all nine activities — this section ties everything back to the notation your students will meet in their textbook.

M = [ [a, b], [c, d] ]  →  î lands at (a, c), ĵ lands at (b, d)
M · (x, y) = (a·x + b·y, c·x + d·y)
det(M) = ad − bc  ·  M⁻¹ = (1/det) · [ [d, −b], [−c, a] ]
